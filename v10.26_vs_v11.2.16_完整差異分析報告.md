## 一、架構差異

### 1.1 文件結構

| 項目 | v10.26 | v11.2.16 |
|------|--------|----------|
| 總文件數 | 1 個（單文件） | 30+ 個 |
| CSS | 內聯在 HTML `<style>` 中（約 800 行） | 拆分為 `css/*.css` 5 個文件，build.js 合併 |
| JS | 內聯在 HTML `<script>` 中（約 6000 行） | 拆分為 `js/**/*.js` 23 個文件，build.js 合併 |
| 構建方式 | 直接編輯 HTML 文件 | 修改模塊源文件 → 執行 `node build.js` → 生成 `dist/index.html` |
| 部署方式 | 手動上傳到伺服器 | GitHub push → Railway 自動部署 |
| 測試 | 無 | `test.js` 44 個自動回歸測試（build.js 自動執行） |

**影響**：
- v11 修改後**必須**執行 `node build.js`，否則改動不會出現在 `dist/index.html`
- v11 部署依賴 GitHub + Railway，任何環節出錯都會導致部署失敗
- v11 模組化使代碼更易維護，但模塊間依賴問題更難排查

### 1.2 版本檢測與快取清除機制

| 項目 | v10.26 | v11.2.16 |
|------|--------|----------|
| Service Worker | ✅ 強制解除所有 SW + 清除 Cache API | ❌ 完全無 SW 處理 |
| 版本檢測 | 無（依賴 SW 解除） | ✅ localStorage `macau_app_version` 檢測 |
| 快取清除觸發 | SW 解除 + Cache.delete() | 版本變更時 `location.replace(url + '?_v=' + Date.now())` |
| 版本顯示 | 硬編碼 `window.APP_VERSION = 'v10.26'` | 動態 `var APP_VERSION = 'v11.2.16'`（自動比對 localStorage） |

**影響**：
- v11 的版本檢測**不如 v10.26 徹底**：v10.26 會強制清除所有瀏覽器 Cache API，v11 只依賴 URL 參數和 localStorage
- 如果瀏覽器有舊版 Service Worker 殘留，v11 **無法清除**，可能導致快取舊版 HTML

### 1.3 伺服器架構（v11 新增）

| 項目 | v10.26 | v11.2.16 |
|------|--------|----------|
| 後端伺服器 | 無（純靜態 HTML） | ✅ `server.js`（Express） |
| Firebase 同步 | 直接在瀏覽器中使用 Firebase SDK | 同樣在瀏覽器中，但透過 `server.js` 提供 HTML |
| HTTP 快取頭 | 無（靜態託管） | `server.js` 設定 `Cache-Control`（曾導致快取問題，v11.2.6 修復） |
| 健康檢查 | 無 | ✅ `/health` API（回傳版本號） |

**影響**：
- v11 引入 `server.js` 後，增加了一個潛在故障點（Railway 部署失敗的原因）
- v10.26 是純靜態文件，可以直接放在任何靜態託管服務；v11 需要 Node.js 運行環境

### 3.2 函數實作差異（重要）

以下函數在 v10.26 和 v11.2.16 中**都存在，但實作不同**：

#### `calcComm(vol, rate)`
- v10.26: `return Math.round(vol * rate * 100) / 100;`
- v11.2.16: 相同（經 `test.js` 驗證）
- **影響**：無

#### `calcFund(comm, bonus)`
- v10.26: `return Math.round((comm - bonus) * 100) / 100;`
- v11.2.16: 相同（經 `test.js` 驗證）
- **影響**：無

#### `fmt(n)`
- v10.26: 有處理 `n >= 1e8` 的格式
- v11.2.16: 相同（經 `test.js` 驗證）
- **影響**：無

#### `toNum(s)`
- v10.26: 有處理 `"50万"` 格式
- v11.2.16: 相同（經 `test.js` 驗證）
- **影響**：無

#### `saveData(silent)`
- v10.26: 直接存到 `localStorage`
- v11.2.16: 相同（但可能有 Firebase 同步邏輯差異）
- **影響**：需測試 Firebase 同步是否正常

#### `loadData(silent)`
- v10.26: 直接從 `localStorage` 讀取
- v11.2.16: 相同（但可能有 Firebase 同步邏輯差異）
- **影響**：需測試 Firebase 同步是否正常

### 3.3 模塊化後的函數掛載問題

v10.26 所有函數都是**全域的**（直接定義在 `window` 下）。
v11.2.16 函數定義在模塊中，需要透過 `window.* = *` 掛載。

**潛在問題**：如果某個函數沒有正確掛載，呼叫時會報 `undefined`。

從 Grep 結果來看，v11.2.16 的主要函數都有掛載（因為系統能運行），但需逐一確認：
- `RM` 物件：`window.RM = RM` ✅
- `hc*` 函數：直接在 `hotel-config-logic.js` 定義，應該是全域的 ✅
- `renderRankChart()`：`window.renderRankChart = renderRankChart` ✅
- `_startLoginParticles()`, `_stopLoginParticles()`：**需確認** ⚠️

## 六、v11 引入的 Bug 清單

### 6.1 CryptoJS 缺失（v11.0 ~ v11.2.6）

- **發生版本**：v11.0（模組化時遺漏）
- **發現版本**：v11.2.6
- **問題**：`js/crypto.js` 的 `encryptData()`/`decryptData()` 拋 `CryptoJS is not defined`
- **根因**：`template.html` 從 v11.0 創建時就沒有包含 CryptoJS CDN 腳本（舊版單文件有）
- **影響**：加密功能失效，JSON 不加密存儲（被 try-catch 兜住，靜默降級）
- **修復**：v11.2.6 在 `template.html` Firebase SDK 後補回 CryptoJS CDN
- **狀態**：✅ 已修復

### 6.2 #hc-modal-bg CSS 缺失（v11.0 ~ v11.2.15）

- **發生版本**：v11.0（模組化時遺漏）
- **發現版本**：v11.2.15
- **問題**：`#hc-modal-bg` 沒有 `display:none` + 沒有 `.hc-modal` 樣式
- **根因**：v11 模組化時遺漏了這段 CSS（約 95 行）
- **影響**：酒店配置彈窗在頁面加載時直接顯示（沒有隱藏）
- **修復**：v11.2.15 在 `components.css` 末尾補上完整的 `#hc-modal-bg`、`#hc-help-bg`、`#hc-quick-bg`、`.hc-modal` 等 CSS
- **狀態**：✅ 已修復（但後來發現不是真正根因，真正根因是 #6.3）

### 6.3 #page-room CSS 權重 bug（v11.1.1 ~ v11.2.16）

- **發生版本**：v11.1.1（修復房務系統桌面端佈局時引入）
- **發現版本**：v11.2.16（今天）
- **問題**：`#page-room { display: flex; }` 權重過高（ID 選擇器），覆蓋 `.page { display: none; }`（class 選擇器）
- **根因**：v11.1.1 修復房務系統桌面端佈局時加入，但**忘了加上 `.active` 條件**
- **影響**：**極大** — 房務系統頁面永遠顯示，不管是否在房務頁面（用戶反映「修改了3天還在」）
- **修復**：v11.2.16 改為 `#page-room.active { display: flex; }`
- **狀態**：✅ 已修復（v11.2.16 已 push 到 GitHub，等待 Railway 部署）

### 6.4 build.js 在 Railway 崩潰（v11.2.5 ~ v11.2.8）

- **發生版本**：v11.2.5（某次修改 `package.json` 後）
- **發現版本**：v11.2.8
- **問題**：`build.js` 跑 `test.js` → 需要 `jsdom` → Railway 沒有 → 崩潰
- **根因**：Railway 的 `package.json` 僅包含 `express` 依賴，沒有 `jsdom`；`npm start` = `node build.js && node server.js` → 崩潰 → `server.js` 永遠啟動不了
- **影響**：**極大** — 所有 Railway 部署全部 FAILED
- **修復**：v11.2.8 加入環境檢測：`process.env.NODE_ENV !== 'production' && !process.env.RAILWAY_ENVIRONMENT` → 生產環境跳過測試
- **狀態**：✅ 已修復

### 6.5 server.js Cache-Control 覆蓋 HTML meta（v11.2.6）

- **發生版本**：v11.2.6（修改 `server.js` 時）
- **發現版本**：v11.2.6
- **問題**：`server.js` 的 HTTP 響應頭 `Cache-Control: public, max-age=300` 覆蓋 HTML `<meta>` 標籤
- **根因**：HTTP 響應頭優先級高於 HTML meta 標籤
- **影響**：瀏覽器快取策略失效，舊版 HTML 可能被快取
- **修復**：v11.2.6 修改 `server.js` 的 Cache-Control 為 `no-cache, no-store, must-revalidate`
- **狀態**：✅ 已修復

### 6.6 根目錄舊 index.html 殘留（v11.2.14 發現）

- **發生版本**：v11.0（模組化時）
- **發現版本**：v11.2.14
- **問題**：舊 `index.html`（v11.2.2）在根目錄，`server.js` 會 fallback 到它
- **根因**：`server.js` 有 fallback 邏輯：`if (!fs.existsSync(dist/index.html)) → use root index.html`；Railway 的 `package.json` start script 是 `rm -rf dist && node build.js`，但 `build.js` 崩潰（見 #6.4），所以 `dist/index.html` 永遠不會被創建
- **影響**：**極大** — Railway 一直服務舊版 v11.2.2，即使用戶看到 v11.2.13 部署成功
- **修復**：v11.2.14 刪除舊 `index.html`，簡化 `package.json` start script 為 `node server.js`
- **狀態**：✅ 已修復
