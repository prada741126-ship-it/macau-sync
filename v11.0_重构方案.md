## 三、技术方案

### 3.1 构建系统（零依赖）

使用自定义 Node.js 构建脚本 `build.js`，**不需要 npm 安装任何依赖**：

```
build.js 流程:
1. 读取 version.json → 生成构建号
2. 按依赖顺序读取所有 CSS 文件 → 合并为一个 <style> 块
3. 按依赖顺序读取所有 JS 文件 → 合并为一个 <script> 块
4. 将合并后的 CSS + JS 注入 index.html 模板
5. 注入构建信息（版本号、时间戳）作为 HTML 注释
6. 输出到 dist/index.html

一次性命令: node build.js
```

### 3.2 版本管理

```json
// version.json
{
  "version": "11.0.0",
  "buildDate": "2026-06-08",
  "buildNumber": 1,
  "description": "模块化重构"
}
```

构建后自动注入到 HTML：
```html
<!-- BUILD: v11.0.0 (2026-06-08 #001) -->
```

### 3.3 缓存策略

**移除暴力清缓存**，改用正确方式：

| 问题 | 旧方案 | 新方案 |
|------|--------|--------|
| Service Worker | 每次强制解除 | 移除所有 SW 注册代码 |
| Cache Storage | 每次清除 | 移除清除逻辑 |
| 版本更新 | 无感知 | version.json → 构建号 → 页面底部显示版本 |
| 缓存控制 | 依赖浏览器 | Railway 静态服务器配置 `Cache-Control` |

### 3.4 类型 / Schema 验证

引入轻量级 JSON Schema 验证（写在 `config.js` 中，**无外部依赖**）：

```javascript
// 交易数据 schema
const TX_SCHEMA = {
  id: 'number', date: 'string', agent: 'string', venue: 'string',
  volume: 'number', rate: 'number', comm: 'number', bonus: 'number',
  drawn: 'number', fund: 'number', type: "'rolling'|'cash'"
};

function validateTx(tx) { /* 运行时校验 */ }
```

## 五、风险控制

### 5.1 回退策略

```
index.html                  ← 当前 v10.26（保留不动）
index_v10.26_GOLDEN.html    ← 重构前复制一份
dist/index.html             ← v11.0 构建产物（部署用）
```

任何时候出错 → 直接部署 `index.html`（v10.26）即可回退。

### 5.2 渐进式迁移

- **Phase 1 的构建脚本**是最关键的基础设施
- 每个 Phase 结束时运行 `node build.js` 验证
- 每完成一个 Phase 就提交 git + 部署测试
- 不删原 `index.html`，只标记为 GOLDEN

### 5.3 构建验证清单

每个 Phase 完成后执行：
```bash
node build.js                    # 构建
node -e "require('fs').read..."  # 语法检查
# 手动测试核心功能
```
